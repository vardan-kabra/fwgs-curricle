# FWGS Curricle

Live, hosted versions of the Fountainhead School **Meal Break Finder** and **Bus Stop Finder**.
A landing page links to two standalone dashboard pages. Each dashboard fetches its data
live from a private Google Sheet (via a small Apps Script endpoint) instead of having the
data baked into the HTML.

## How it works (architecture)

```
Static site (any host: Cloudflare Pages / GitHub Pages / Netlify)
├── index.html      landing page → links to the two dashboards
├── meals.html      Meal Break Finder  (UI unchanged; data fetched live)
├── buses.html      Bus Stop Finder    (UI unchanged; data fetched live)
├── assets/
│   ├── config.js      ← the ONE file you edit to go live (endpoint URL)
│   └── live-data.js   fetch + cache + status/error UI
└── fixtures/
    ├── meals.json     snapshot / local mock / contract reference
    └── buses.json
            │  fetch ?sheet=meals / ?sheet=buses
            ▼
Apps Script web app (apps-script/Code.gs) — reads the two PRIVATE spreadsheets
   by ID and returns JSON. Only the script can read the sheets.
```

### Decisions (the "why")

- **Static frontend + Apps Script JSON endpoint** (not a pure Apps Script web app):
  the pages paint instantly from a CDN; only the data fetch touches Google, so the
  Apps Script cold-start (2–5 s) is hidden behind a loading state instead of a blank page.
  Sheets stay private; dashboards are open to anyone with the link.
- **Stale-while-revalidate caching** (`assets/live-data.js`): a returning visitor sees
  last-known data immediately, then it refreshes in the background. If Google is
  unreachable, the page keeps showing the cached copy with a quiet
  "showing saved copy · updated X ago" note + Retry. Only a first-ever visit with no
  cache and no network shows a hard error.
- **`getDisplayValues()` server-side**: the dashboards print time strings ("8:40 am")
  verbatim, so the endpoint must return the sheet's *formatted* text, not Date objects.
- **The dashboards are generated, not hand-edited** — see `tools/build-pages.mjs`.

## The data contract

`fixtures/*.json` define the exact JSON the endpoint must return. They are the source of
truth for both the local mock and the Apps Script transform.

- **meals**: `{ groups:[…order…], data:[ {id,label,group,events:[{service,time,sort,location,sub?}]} ] }`
  — `service` ∈ `morning | lunch | pantry | exit`.
- **buses**: `[ {id,badge,direction,who,schoolEvent,schoolTime,timeWord,routes:[{name,stops:[{name,code,time}]}]} ]`
  — `badge` ∈ `Morning | Afternoon`, `direction` ∈ `To School | Going Home`.

## Local development

```bash
node tools/serve.mjs          # serve on http://localhost:4321 (uses fixtures)
# or use the Claude Code preview ("fwgs" launch config)
```

With `assets/config.js` → `endpoint: ""`, pages load `fixtures/*.json`.

## Regenerating from the originals

```bash
node tools/extract-fixtures.mjs   # re-derive fixtures/*.json from the Downloads originals
node tools/build-pages.mjs        # re-derive meals.html / buses.html (asserts every edit)
```

### Sheet conventions the endpoint relies on

- **Meals** come from a dedicated **`Dashboard Data`** tab (generated by
  `tools/gen-meal-tab.mjs`), not the human poster — the poster's grade column
  (`1 & 6` = grades 1–6, but `7 & 9` = only 7 and 9; `7 TO 12` = 7–10 in the MYP
  row but 11–12 in the DP row) is too ambiguous to parse safely.
- **Buses**: each trip is its own tab, identified by the time token in the tab
  name (740 / 905 / 1210 / 210 / 345). Stops are the rows between a `Route X`
  header and the `Fountainhead School` row; time is **column E** with the
  meridian taken from the trip (AM for morning tabs, PM for afternoon). Raw stop
  names are lightly normalized (trailing periods, double spaces, `:-` → `.`).

## Going live

1. **Meal data tab** — `node tools/gen-meal-tab.mjs`, then in the MEAL
   spreadsheet create a tab named exactly **`Dashboard Data`** and import
   `out/meal-dashboard-tab.csv` (File → Import → *Insert new sheet*, then rename
   if needed). The poster stays untouched for humans; the endpoint reads only
   `Dashboard Data`, and future meal edits go there.
2. **Endpoint** — deploy `apps-script/Code.gs` (see its header). Run `ping` once
   to grant the Sheets prompt, then Deploy → Web app (Execute as: Me,
   Access: Anyone). Copy the `/exec` URL.
3. **Config** — set `endpoint` in `assets/config.js` to that URL.
4. **Host** — upload `index.html`, `meals.html`, `buses.html`, `assets/` (and
   `fixtures/` as the offline fallback). `tools/`, `apps-script/`, `out/` and the
   originals are not needed at runtime.
