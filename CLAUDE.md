# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository shape

One GitHub repo (`fwgs-curricle`), two **independent static apps** under it, each self-contained with its own `assets/` and `tools/serve.mjs`:

- **`school-dashboards/`** — live Meal Break Finder + Bus Stop Finder. Static frontend that fetches JSON from a Google Apps Script web app reading private Google Sheets.
- **`parent-hub/`** — parent-facing pages that embed Google Drive PDFs via a slug-based registry. Planned split: **`index.html`** (admin/logistical — uniform, food, transport, academic calendars, communication, contacts) and a future **`academics.html`** (academic content — IB policies, programme brochures, programme info, Google-Site programme-page links). `test-embed.html` is a sandbox for the embed pattern. Working spec for `index.html` is at `parent-hub/SPEC.md`; pending data and planned enhancements (comms-app links, the Who-to-Contact organogram, etc.) are tracked in `parent-hub/TODO.md` — the live backlog.

No build step, no `npm install`, no bundler. Pages are plain HTML loading vanilla JS that exposes globals (`window.FWGS_CONFIG`, `window.FWGS_RESOURCES`, `window.FWGSLiveData`, `window.FWGSResources`). Editing a JS file in an `assets/` folder IS the deploy step.

Both apps deploy to **Cloudflare Pages** from the `main` branch root — **auto-deploy on every push** (a Pages project connected to the GitHub repo; no build step, build output = repo root). Live URL pattern: `https://fwgs-curricle.pages.dev/<app>/<page>.html` — this is the URL the parent-facing Google Site embeds. (GitHub Pages at `vardan-kabra.github.io/fwgs-curricle/...` was the original host and may still be enabled as a fallback. A Cloudflare **Worker** + Git was tried first and abandoned — a Worker needs a `wrangler` config to deploy; Pages serves the static files with zero config.)

## Common commands

```bash
# Preview servers (also wired into .claude/launch.json — preview_start "fwgs" / "parent-hub")
node school-dashboards/tools/serve.mjs    # → http://localhost:4321
node parent-hub/tools/serve.mjs           # → http://localhost:4322

# school-dashboards data pipeline (run from school-dashboards/)
node tools/extract-fixtures.mjs           # rebuild fixtures/*.json from Downloads originals
node tools/build-pages.mjs                # regenerate meals.html / buses.html (asserts every edit)
node tools/gen-meal-tab.mjs               # build out/meal-dashboard-tab.csv to paste into the meals sheet
node tools/validate-meals.mjs             # validate meals fixture against spec
node tools/validate-buses.mjs             # validate buses fixture against spec

# Publish to live site
git add -A && git commit -m "..." && git push    # Cloudflare Pages auto-deploys in ~1 min
```

Each `serve.mjs` resolves its own root as `..` from its location, so server for one app cannot serve files from the other — they are isolated by design.

## school-dashboards architecture

```
Browser → static page → assets/live-data.js
                              ↓ fetch(?sheet=meals|buses)
                       Apps Script web app (apps-script/Code.gs)
                              ↓ openById + getDisplayValues
                       Private Google Sheets
```

**The one file you edit to go live:** `school-dashboards/assets/config.js` — set `endpoint` to the Apps Script `/exec` URL. With `endpoint: ""` the pages load `fixtures/*.json` instead (this is the local-dev path AND the offline fallback).

**Stale-while-revalidate** (`assets/live-data.js`): cached payload paints immediately, network refresh updates it. If Google is unreachable, page keeps the cached copy with a "showing saved copy" pill + Retry. Only a first-ever visit with no cache and no network shows a hard error. `onData(payload)` may be called twice (cache, then network) — it must be idempotent.

**Data contract = `fixtures/*.json`.** These define the exact JSON shape the endpoint must return; both the local mock and the Apps Script transform target them.
- `meals.json`: `{ groups:[…order…], data:[ {id,label,group,events:[{service,time,sort,location,sub?}]} ] }` — `service` ∈ `morning | lunch | pantry | exit`.
- `buses.json`: `[ {id,badge,direction,who,schoolEvent,schoolTime,timeWord,routes:[{name,stops:[{name,code,time}]}]} ]` — `badge` ∈ `Morning | Afternoon`, `direction` ∈ `To School | Going Home`.

**`meals.html` and `buses.html` are generated, not hand-edited** — see `tools/build-pages.mjs`. Hand-edits get overwritten on the next build.

**Apps Script gotchas:**
- Must use `getDisplayValues()` (not `getValues()`) — dashboards print time strings verbatim ("8:40 am"), not Date objects.
- Meals are read from a dedicated **`Dashboard Data`** tab generated by `tools/gen-meal-tab.mjs`, NOT the human-readable poster (whose grade ranges like `7 & 9` vs `7 TO 12` are too ambiguous to parse).
- Bus tabs are identified by time token in tab name (`740`, `905`, `1210`, `210`, `345`); check `1210` before `210`.
- Don't add custom request headers from the client — a plain GET is CORS-simple; custom headers trigger a preflight that Apps Script can't answer.
- Server-side cache via `CacheService` is 10 min (`CACHE_SECONDS = 600`).

**Deploying the endpoint:** paste `apps-script/Code.gs` + `appsscript.json` into a new Apps Script project, run `ping` once to grant the Sheets prompt, Deploy → Web app (Execute as: Me, Access: Anyone), copy `/exec` URL into `assets/config.js`.

## parent-hub architecture

```
HTML page (e.g. test-embed.html)
   ↓ <a data-resource="uniform-policy">       ← references slugs, not URLs
   ↓ <iframe data-resource="..." data-mode="embed">
assets/resources.js
   ├── window.FWGS_RESOURCES = { slug: { title, type, driveId|url } }   ← the registry
   └── window.FWGSResources  = { url(), embed(), title(), wire() }      ← helpers
       └── wire() runs on DOMContentLoaded, sets href/src from registry
```

**The one file you edit to change/add Drive PDFs or external links:** `parent-hub/assets/resources.js`. Store the Drive **`driveId`** (not the full URL) — Drive URL schemes have changed before; centralising URL-building in helpers means a Drive format change is a one-file fix.

Entry types:
- `"drive-pdf"` → `url()` returns `/file/d/{id}/view`, `embed()` returns `/file/d/{id}/preview`
- `"drive-image"` → `url()` returns `/file/d/{id}/view`, `embed()` returns `/uc?export=view&id={id}`
- `"url"` → returns the literal `url` field

To verify Drive PDF embeds, the file must be shared as **"Anyone with the link can view"** — otherwise the iframe shows a Drive sign-in wall instead of the document.

The `assets/config.js` and `assets/live-data.js` in `parent-hub/` are copies of the school-dashboards versions; they are unused by `test-embed.html` but available if a parent-hub page later wants live sheet data.

**The admin hub (`index.html`) is spec-driven** — `parent-hub/SPEC.md` is the canonical structure doc. Read it before editing `index.html`. Key patterns it establishes:
- **Single source of truth:** each topic has exactly one home section; cross-refs are anchor links, not duplicated content.
- **Build-now-hide-now:** sections/tiles for content that's pending (no data yet) ship in the HTML wrapped in `<... data-status="pending">` + a global CSS rule `[data-status="pending"] { display: none; }`. To enable later, remove the attribute — no structural change needed.
- **Content from external specs** (e.g., the user's separate Communication Build Spec for Section 3) goes straight into the rendered HTML; we don't copy the source spec into the repo if it contains internal/operational notes marked "do not render."

## Conventions / gotchas

- **`.claude/launch.json` is committed** (shared preview server configs); **`.claude/settings.local.json` is gitignored** (per-user).
- **Pages on mobile (≤768px) hide PDF iframe embeds** via `#pdf-embeds { display: none }` — the link-list section remains so parents can still open PDFs. Use this same pattern for future parent pages embedding Drive PDFs.
- **Add `<meta name="robots" content="noindex, nofollow">`** to parent-hub pages that should not appear in search results (the public URL exists but isn't meant to be the entry point — Google Sites is).
- **Always pause for explicit confirmation before `git commit` / `git push`** — and never use `git add -A` blindly. Survey `git status` first; the photo folders are ~120 MB and easy to over-stage.
- **Filenames with spaces work** in the HTML (`assets/brand/FWGS - SVG Logo.svg`) because modern browsers URL-encode automatically. Don't rename existing assets just to remove spaces.
- **LF/CRLF warnings on git commit are expected on Windows** and harmless — git's `core.autocrlf` normalizes on checkin.
